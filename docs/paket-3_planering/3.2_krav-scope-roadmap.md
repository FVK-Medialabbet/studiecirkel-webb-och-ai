## Mål

Efter den här modulen ska du:

- förstå skillnaden mellan ”måste”, ”bör” och ”kul sen”
- kunna skriva enkla **user stories** på vardagsspråk
- kunna göra en **första roadmap** med små delversioner

## Innehåll

### Krav och scope – vad ska vara med nu, vad kan vänta?

Utgå från idé och målbild från 3.1 och lista:

- Vad *måste* finnas i första versionen?
- Vad *bör* finnas om det finns tid?
- Vad är ”kul sen” om projektet lever vidare?

Du kan använda en enkel variant av MoSCoW utan facktermer:

- **Nu** – absolut viktigast
- **Snart** – viktigt men kan vänta
- **Senare** – nice-to-have

### User stories på enkel svenska

En user story beskriver **vem** som gör **vad** och **varför**:

> ”Som [typ av användare] vill jag [göra något] så att jag kan [nytta].”

Exempel:

- ”Som medlem vill jag kunna se alla kommande aktiviteter så att jag kan planera min vecka.”
- ”Som aktivitetsledare vill jag kunna lägga till en ny aktivitet så att deltagarna ser den i listan.”

Skriv hellre få och tydliga stories än många och otydliga.

### Enkel roadmap

Dela upp projektet i små versioner, t.ex.:

- **v0.1** – absolut minsta som behövs för att du själv ska kunna testa
- **v0.2** – så att en liten grupp användare kan prova
- **v0.3** – så att du kan visa upp det för fler

Koppla dina user stories till versioner:

- vilka stories ska ingå i v0.1?
- vilka kan vänta till v0.2 eller v0.3?

## Förslag på självstudier

- Skriv 3–7 user stories för ditt projekt.
- Sortera dem i ”nu/snart/senare”.
- Försök rita upp en enkel roadmap med v0.1, v0.2, v0.3 och skriv under varje version:
  - vilka stories som hör dit
- Om du vill:
  - använd GitHub Issues och skapa en issue per user story
  - märk dem med etiketter ”nu”, ”snart”, ”senare” eller version.
